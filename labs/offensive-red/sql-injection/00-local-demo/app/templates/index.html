SQLi Cheat Sheet (Training)

Use this block to explain what each injection style tries to do and how defenders should respond.

Injection type Goal Where to test in this lab Example training payload Expected behavior Defender takeaway
Boolean-based (OR logic) Force condition to evaluate true. Demo Login username field ' OR '1'='1' -- In vulnerable: may bypass auth. In safe: login should fail. Use parameterized queries and strict auth checks.
Union-based probing Try to merge attacker-controlled results. Product Search and Demo Login ' UNION SELECT 1,2,3,4 -- In vulnerable: may show SQL errors or mixed rows. In safe: treated as plain text. Hide detailed DB errors and monitor union/comment markers.
Comment truncation Ignore part of original SQL query. Demo Login payload endings admin' -- In vulnerable: query logic can be altered. In safe: no SQL execution effect. Detect repeated comment patterns and correlate with auth failures.
Error-based feedback Learn query structure from DB errors. Demo Login and Product Search ' UNION SELECT 1,2 -- (column mismatch demo) In vulnerable: SQL errors can leak structure hints. Return generic app errors; log details server-side only.

Training only: run payloads in this local lab environment and compare the same input in both modes.

What each mode does

  • Vulnerable mode: the app trusts what you type and inserts it directly into SQL code.
  • Safe mode: the app treats your input as plain text data, so it cannot become a SQL command.
  • Monitoring board: tracks suspicious markers, action type, and event timeline for SOC triage practice.
